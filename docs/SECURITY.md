# Security and Safety Design

## Trust model

UtilityForge is a local desktop application processing user-selected files, folders, text, and external command output. Inputs are untrusted even when selected by the user: names can contain control characters or option-like prefixes, media can be malformed, folders can contain symlinks, and external executables can behave unexpectedly.

The application is not a sandbox. External codecs and archive tools process content with the user's privileges. The design therefore minimizes ambient work, limits what is invoked, and never elevates privileges.

## Core rules

- Never invoke a shell for an operation. Start a resolved executable and pass a `QStringList` of distinct arguments to `QProcess`.
- Never modify an input in place. Write into an application-owned staging file in the chosen destination and publish only after success.
- Never request root privileges or launch a package manager.
- Accept local file URLs only in version 0.1. Reject remote, malformed, and unsupported URL schemes.
- Validate operation, input type, output directory, tool capability, free-space assumptions where practical, and collision policy before launch.
- Treat paths as opaque Unicode values. Do not interpolate them into commands, markup, rich text, or log formatting.
- Bound memory, captured logs, QR input length, and the amount of UI data retained.
- Do not follow symlinks implicitly when calculating or displaying scope; archive behavior must be explicit and tested.

## External process safety

`ExternalToolRegistry` resolves each supported program using `QStandardPaths::findExecutable`, stores an absolute path for the session, and probes it with a bounded, known command. An executable must match the expected probe before it becomes available. The configured program is never taken from dropped content.

`ProcessRunner` owns each `QProcess` and applies the following policy:

- program and arguments are separate; shell metacharacters have no special meaning;
- paths are normalized to absolute paths before use, while preserving the selected object rather than silently substituting an unrelated target;
- a deliberate working directory is set, especially for archive member names;
- the process environment is constructed from a controlled copy of the user environment, removing variables known to inject behavior into loaders or tools where practical;
- stdout and stderr are read incrementally through signals and stored in bounded buffers;
- machine-readable output is used when offered, such as FFprobe JSON and FFmpeg progress records;
- startup and probe timeouts are enforced asynchronously;
- no `waitForStarted`, `waitForFinished`, or blocking pipe read occurs on the GUI thread;
- cancellation sends `terminate()`, waits via a timer for a short grace period, then calls `kill()` if necessary;
- exit status, exit code, expected output type, and output existence are all checked before success is reported.

Tool arguments generated by the application are allowlisted. Version 0.1 has no “custom FFmpeg arguments” field and does not execute commands copied from the clipboard.

## Output integrity

Each task receives an unpredictable, application-owned staging name in the final output directory where possible. Same-filesystem placement allows a successful result to be renamed into place without cross-device partial copies. The final destination is never exposed as complete until validation passes.

On cancellation or failure, UtilityForge deletes only the exact staging paths created and recorded by that task. It never recursively deletes a user-selected input or an unresolved path. If cleanup fails, the error names the leftover staging file so the user can inspect it.

Output collisions default to creating a numbered unique name. Replacing an existing file requires a separate explicit confirmation and a recoverable strategy; replacement can remain postponed for 0.1. Source and destination identity is checked before starting.

For image conversion performed in-process, the decoder runs off the GUI thread. Output uses `QSaveFile` or an equivalent staged commit. Pixel dimensions are checked before full decode to reduce decompression-bomb risk, with a documented maximum pixel count and an override only if later justified.

## Archive safety

Version 0.1 creates archives but does not extract them, eliminating path-traversal-on-extraction risk. Inputs are passed relative to an explicit parent working directory so archives do not disclose absolute paths. Symlinks are archived as links rather than followed; the preflight summary states this. Special files, sockets, devices, and inaccessible entries cause the task to fail, so an incomplete archive is never silently published.

Archive member names are display-escaped in logs. Newlines and control characters in filenames must not forge log entries.

## Media safety

FFprobe and FFmpeg receive only selected local inputs and staged outputs. Probe output has byte and time limits. Remux/re-encode commands use application-owned presets; stream selection is based on validated JSON, not stderr scraping. Version 0.1 rejects unsupported data streams, attachments, subtitles, and extra audio/video streams rather than dropping them.

FFmpeg is a high-risk parser of untrusted media, so distributions should provide security updates. UtilityForge reports the resolved binary and version in diagnostics and does not bundle an unmaintained copy by default.

## QR and clipboard safety

QR generation treats text as data only. It does not open, fetch, validate, or execute encoded URLs. Input length is bounded and rendering arithmetic is overflow-checked.

Clipboard actions occur only after a direct user command. Copied files use MIME URLs and copied paths use plain text. The interface explains that some Wayland clipboard providers may not retain data after the application exits. Clipboard contents and source file contents are never written to logs.

## Settings, logs, and privacy

Settings contain UI preferences, safe output defaults, and tool discovery results suitable for caching. They must not contain secrets. Logs are local, bounded, and rotated or session-scoped. They omit file contents, environment dumps, clipboard data, and credentials; paths may be present because they are essential to local diagnostics, so the UI warns before a user shares a log.

No telemetry, analytics, update checker, or network call exists in 0.1. Later GitSnap networking requires a separate threat review covering credentials, hostile repositories, URLs, protocol handling, and editor launching.

## UI and notification safety

Untrusted strings are rendered as plain text. Rich-text labels are disabled for paths, tool output, and error messages. User-facing errors are structured and concise; technical details are available separately and cannot contain clickable commands by default.

Notification actions refer to an internal task/result identifier, not an arbitrary path or command embedded in the message. Opening an output uses the recorded successful result and `QDesktopServices`.

## Security verification

Tests must cover:

- spaces, quotes, leading dashes, newlines, Unicode, and shell metacharacters in filenames;
- malicious or oversized FFprobe output and failed tool probes;
- cancellation at startup, during output, and immediately before publish;
- symlinks, special files, permission failures, collisions, and cleanup failures;
- image dimension limits and corrupt files;
- process exit success with missing/invalid output;
- log truncation and plain-text rendering;
- environment sanitization and the guarantee that no shell is invoked.

Release builds should be exercised with compiler warnings elevated in CI, AddressSanitizer and UndefinedBehaviorSanitizer test jobs, and dependency/security update monitoring appropriate to Fedora packaging.
